  • to create a git repo - "git init"
  • Create a new branch - "git checkout -b [branch-name]
  • Pull the lastest verison on the origin - "git pull"

Saving changes

  • Stage Changes - "git add ."
  • Commit Changes - "git commit -m 'message-of-what-changed'"
  • Save the changes on github - "git push origin 'branch-name-changes-are-on'"

I told myself I would only make that joke once in this guide - lets see if I hold to that. Before we get too far into this, we need to go over a few basic terms, and what they mean

  • Respository - A collection of files and directories you are monitoring for changes using a version control tool. Sometimes shortened to repo.
  • Hash - A string of characters that acts like a digital fingerprint for a file or portion of a file. The type of hash used by git is the SHA1 cryptographic hash.
  • Graph - A diagram of nodes connected by lines called edges.
  • DAG - Directed Acyclic Graph. A graph were the edges are arrows (directed) and no matter which arrow you follow you can never travel in a circle (acyclic).
  • CLI - Command Line Interface. The “Terminal” or “Shell” of your operating system. Git is a CLI tool but there are also graphical (GUI) interfaces.

Initializing a Local Respository


In order to get started, you need to be working in a Respository that has been "initialized". In order to initialize a respository, just type

git init


after which, you can check the status of the repo for changes, or additions, at any point with

git status


and you can compare the differences between the current files and the last commit with

git diff

Adding a remote origin

When you think of Git, you probably think of Github - a site to store your respository so you can pull them down on different machines to work on them (also, as a secure backup - trust me, you dont want a hard drive failure ruining your final project) In order to set up your local repository with a remote one (AKA, on GitHub), you then need to do the following

To save your git repository to your GitHub account, create a new repository by way of the “create a new repo” button in the top corner of the GitHub website. IMPORTANT: Be sure not to click the checkbox to add an initial README file.

Let’s say you’ve created a new repo called “rails_project” and you’ve already initialized this project’s git repository. From the command line (within your project folder) add Github as a remote:

git remote add origin git@github.com:[username]/rails_project.git

note - This should be all one line with [username] replaced by your actual username.


When ever you want to push the latest state of your repo to Github:

git push origin [branch-name]

where the branch name is the branch that you have changes on


If you create a new repo on github, it will also show you the steps of how to connect that with an initialized repository on your local machine

Let's Branch off from that topic for a second

Branches in git give you a safe space to work on new features. The main branch of your repository is called the master branch, although that is being updated to main


Commits made in other branches can be merged into the master/main at any time. Branches can also be abandoned or even deleted at any time leaving the master unchanged.

What this means is that when developing a new feature, you can create a branch to work on. If the feature goes sideways in a big way, you can just abandon that branch and go back to your last stable main branch to start over without having to untangle whatever changes you may have made on that branch.

Let's go over those commands

  • To create a new branch: git checkout -b new-branch-name
  • To switch branch: git checkout name-of-branch
  • To merge change into the active branch: git merge branch-to-merge
  • To delete a branch: git branch -d branch-to-delete

If you have the repo already set up to work with a remote repository, I tend to "git push origin [branch-name]" and in the console output after that command there will be a link to open a Pull Request. From the Github Documentation "Pull Requests are the heart of collaboration on GitHub. When you open a pull request, you’re proposing your changes and requesting that someone review and pull in your contribution and merge them into their branch. Pull requests show diffs, or differences, of the content from both branches. The changes, additions, and subtractions are shown in green and red." I use that pull request as another chance to go over what I have changed and catch any last minute errors
